# AZ-305 study path

Use this path to practice architecture decisions, tradeoffs and operational requirements rather than memorizing deployed resource names.

## Identity, governance, and monitoring

| Practice task | Evidence | Design discussion |
| --- | --- | --- |
| Diagram GitHub OIDC and GitHub App trust separately | Issuer/audience/subject/scope table | Why passwordless Azure auth does not remove authorization design |
| Review roles for platform, deploy and runtime identities | Access matrix | Least privilege, management/data plane, break glass/PIM |
| Map request, plan and Azure policies | Guardrail map | Where deny, audit, remediation and exception handling belong |
| Review logs, heartbeat and evidence retention | KQL/workbook sketch | Central monitoring, retention, immutability and privacy |
| Compare budget alerts and TTL | Cost-control narrative | Forecast/notification versus enforced lifecycle |

## Compute and application design

Compare:

- App Service for managed code runtime and simple delivery;
- Container Apps for managed container/revision/scale-to-zero;
- AKS for Kubernetes API/extensibility with greater cost/operations.

Write a decision record for a fictional application. Include traffic, scaling, portability, operational skill, networking, identity, deployment strategy, observability, cost and recovery requirements. The answer should not default to AKS merely because it is flexible.

## Reliability and lifecycle

Trace partial creation, scheduler failure, API throttling, stale worker, Azure residual and repository transfer. Explain idempotency, retries, reconciliation, state/inventory separation and fail-closed deletion.

Propose production SLOs for request-to-ready, endpoint availability and expired-environment cleanup. Correctness of irreversible deletion should be an invariant, not merely an availability target.

## Data and integration boundaries

The lab excludes databases and private networking. Extend the architecture on paper:

- choose a data store and backup/DR strategy;
- choose managed identity/data-plane roles;
- decide private endpoint/DNS/egress/ingress design;
- define regional residency and recovery objectives;
- add connection/bootstrap/rotation without exposing secrets to generated repositories.

## Lab-to-production review

Replace or justify each lab choice:

- one disposable subscription;
- public repositories and native public endpoints;
- shared ACR/workspace;
- GitHub Actions request interface;
- dedicated one-node AKS environment;
- 7/90-day state/evidence retention;
- automatic source-repository deletion.

Deliverable: a two-page architecture decision summary with context diagram, compute decision, identity model, lifecycle/DR model, cost controls and prioritized production gaps.
